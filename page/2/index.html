给定一个字符串s，判定，即是否是一个正则表达式的语言。(L将正则表达式映射到语义上)

Lexical Specification

  • 为每一个token类都写一个正则

    • R1（number）：digits+
    • R2（letter）：[a-z]
    • ...

  • 构造一个R，能够将每一个lexeme对应到各个token(s)上

  • 输入：，对，检查：是否属于

    若成功：，说明这一字符串在某个token的正则中；

  • 剔除，然后回到上一个步骤

    上述算法可能出现的问题：

    • 每次选择的子序列可能不唯一。

      对于字符串==，可以选=，也可以选择==。选择哪一个？

      目前的做法：Maximal Munch -- 最长匹配

    • 应该选择哪一种token呢？

      多个token的定义可能重叠：if/else -- 关键词；也可匹配到identifier([a-z][a-z]*)

      此时需要设置优先级。关键词的优先级高于标识符

    • 没有找到匹配呢？

      -- 不允许发生

      定义一个表达错误的正则表达式语法Error，来接受这样的匹配，放在优先级的最后（Error甚至可以包含前述的正确语法，因为优先级最低，不会误判）

    总结：

  • 正则表达式是字符串模式的清晰表达符号

  • 在词法分析中使用正则表达式，需要一些小的扩展

    • 解决歧义：选最长 + 优先级
    • 处理不匹配：定义Error接收错误输入，优先级最低
  • 著名的相关算法
    • 只需要对输入进行单程扫描
    • 每个字符都进行一些操作（查表）

  • 输入字母表

    如果输入到达最后，且到达了接受状态 ==》接收

    否则 --》拒绝

    • 在一个非接收状态结束
    • 卡住 -- >找不到下一个可以转移的方向

FA的语言定义为 接收字符串的集合

DFA：

  • 一个输入，一个状态，一个转移
  • 没有移动

NFA：

  • 一个输入，一个状态，可能有多个转移方向
  • 可以有移动

上述都是NFA，第一个是多个转移；第二个是存在

DFA对于一个输入字符串，只会在状态转移图中使用一条路径（确定）

NFA最终接受，如果其中存在路径是接受的；

DFA较NFA执行更快（选择的不确定性更少）

对同一语言，NFA却可能比DFA小的多。（时间折中空间的折中）

编译原理（二）：词法分析

发表于 2020-05-03 更新于 2021-05-23

Token -- 一些字符串的集合（可以理解为分词）

  • 英语中：名词、动词、形：

  • 编程语言中：token class（看作一类）

  • 编程语言中：

    • Identifier（标识符）：由字母或数字起始，字母或数字组成的一串字符串：A1，Foo，8a
    • Integer（整数）：非空数位字符串集合
    • Keyword（关键字）：else/if/begin
    • WhiteSpace（空格、回车、制表符、换行符...）
    • 标点符号每个都自成一类（操作符除外）

  • 词法分析的任务：

    • 将程序（看作一个长长的字符串）按照他们的角色分成一个个小的串
    • 将这些子串与解析部分通信

    foo = 42 -- [Lexical Analysis] --> <Id, "foo">, <op, "=">, <Int, "42">

    把这些集合再送入解析器

  • 词法分析实现的要点：

    • 识别出一个个子串（称为lexeme）
    • 将每个lexeme在识别到对应的token类

    最终的一个Map实例如<Int, 20>，就是一个token

  • lookahead：向前看

    DO 5 I = 1,25

    DO 5 I = 1.25

    FORTRAN忽略whitespeace，因此前者是25次循环；后者是为变量DO5I赋值为1.25。

    从左向右划线找，识别到DO，并不能确定真的是DO，向前看，看到了“,“后就能够确定是DO而不是某个叫DO5I的变量中的一部分。

    这样的不断向前看（可能会看一个之后还是不确定）使得词法分析变得非常复杂，因而词法分析的一个目标就是尽可能地减少向前看的次数。

  • C++嵌套类型：Foo<Bar<Bar>>中的">>"和输入流的">>"的区分（目前的解决方法是在Bar>后的>前面加一个空格：Foo<Bar<Bar> >

  • 正则表达式的定义：一个作用在上的正则表达式，是一个最小的表达式集合，包含：

    （+：并集；RR：拼接；*：闭包）

    上述定义称为在一个alphabet（字母表）上的grammar语法。

  • 形式语言：Formal Languages

    定义：是字母的集合，从中抽取的字符集合称为一个语言（Language）

    英语字母（Alphabet） -- 英语句子（Language）

    ASCII码 -- C语言程序

    Meaning function L：将句法符号（language）映射到语义上

    L((0+ 1)*) = set of 0's and 1's

    这一映射的意义：

    • 区分syntax和semantics

    • 允许我们单独考虑符号表达的问题（在不同场景下选用不同的表达方法）

    • 表达式和含义不是1-1对应的（多个syntax对应同一个semantics，可以通过等价尝试优化）

      ！！注意是多对一，而不是一对多。也不允许一对多，一对多背后就是歧义。

编译原理（一）

解释器和编译器

  • 编译器：输入是程序代码，输出是可执行文件（从代码的角度，输出是可以执行的机器码，例如Java字节码）
  • 解释器：输入是程序代码、数据data，输出是程序执行结果

所以python可以一行一行执行，C语言必须编译之后才会执行（编译型语言和解释型语言）

SpeedCoding：让程序员编程更快。是一个运行在IBM 701上的解释器（John Backus）

科学计算的要素：浮点计算和数组下标访问（今天再加一个并行）

SpeedCoding优点：

允许程序员直接写浮点计算和下标寻址的指令，底层再设计翻译指令到计算机器码的模块，程序员从手工进行科学计算（手工实现科学计算的机器码）中解放出来，提高编程速度。（3.5+4，原来的做法是程序员直接写3.5、4、求和的机器码，现在直接写3.5+4，有解释器计算这一步的结果）

缺点：

比手工的方式慢很多（10-20倍，今天的解释性语言依然是比编译型语言或者机器码写出的程序慢很多的）

消耗大量内存

进一步研究出FORTRAN I——Formula Translation Program

John Backus认为之前慢的原因是，公式是被解释（Interpreted）的。

如果公式能够先被翻译成机器能够理解的机器码，速度会变快。（不放弃程序员写highlevel程序）

FORTRAN I -- 第一个编译器

FORTRAN I的结构，也是当代编译器的结构

  • 词法分析（Lexical Analysis） -- syntactic

    任务类似于识别一个句子里的各个单词（分词），这里的句子可以理解为一句代码语句。

    token：分割词的时候使用的标记，英语中的空格/标点等

  • 解析（Parsing）-- syntactic

    分析每一个词的词性，进一步向上，得到解析树。

  • 语义分析（Semantic Analysis）--type，scope

    理解句子的意义。困难！人进行词法分析、解析和机器进行的方式是一致的（分词、找词性、判断关系），但理解句子的意思的部分，对机器过于困难。编译器只能做有限的工作，例如寻找句子里的不一致性。

    • 编译器可以寻找变量的绑定（scope）

      他说他妈妈找不到她了。他究竟分别是谁？

      在计算机中，类似的问题：打印的Jack是谁？

    • 编译器可以寻找类型的不匹配

      Jack said her mother left home today. Jack与her显然不匹配；

      计算机中int c = 5.8;同样是一个语义错误，从赋值语句的语法来说，没有问题，但int 和 5.8 类型不匹配。

  • 优化（运行更快，是用更少的内存）

    X = Y * 0和X=0是一样的（在都是int的情况下）

    在浮点情况下：NAN * 0 = NAN

  • 代码生成 -- translation

    将高级语言程序，转换为机器码/汇编语言

Programmer training is the dominant cost for a programming language.

  • 广泛使用的语言发生改变会很慢（教的成本太大）

  • 一个语言的起步是很容易的（没有用户）

  • 新语言更适宜填补现有的空缺

  • 新的语言会趋向于某个老的语言（这样能够减少训练成本）

    Java -- C++

至今没有公认的衡量语言好坏的度量。
